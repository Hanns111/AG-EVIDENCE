# ADR-012: Benchmark PaddleOCR-VL-1.5 como reemplazo del VLM

- **Estado:** BLOQUEADO — incompatibilidad software RTX 5090
- **Fecha:** 2026-03-24
- **Autor:** Claude Code (Auditor)
- **Revisión:** Hans pendiente

---

## Contexto

El pipeline actual (v4.1.0) usa **qwen2.5vl:7b** via Ollama como VLM para extraer
campos de comprobantes escaneados. Rendimiento actual:

| Métrica | Valor |
|---------|-------|
| Tiempo promedio/página | ~28s |
| Tiempo total DIRI2026 (21 páginas VLM) | ~10 min |
| VRAM | ~14.3 GB |
| Modelo | 7B parámetros |
| Precisión campos | Variable (RUC 0%, Fecha 31%) |

**PaddleOCR-VL-1.5** (0.9B, Baidu, enero 2026) reporta:
- 94.5% en OmniDocBench v1.5 (SOTA mundial)
- 8x más pequeño que qwen2.5vl:7b
- Apache 2.0, local-first
- Soporte vLLM + llama.cpp (GGUF)

## Resultados del benchmark — 2026-03-24

### Hardware y software

```
GPU: NVIDIA GeForce RTX 5090 Laptop GPU (24 GB VRAM, sm_120 Blackwell)
OS: WSL2 Ubuntu 22.04
PyTorch: 2.8.0+cu128 (sm_120 = True)
PaddlePaddle: 3.3.0 GPU cu129
PaddleOCR: 3.4.0 + paddlex[ocr] 3.4.2
transformers: 4.57.6
vLLM: 0.18.0
accelerate: 1.13.0
```

### Test 1: PaddlePaddle nativo

| Métrica | Resultado |
|---------|-----------|
| Model load (cached) | 2.7s |
| PDF processing (44 pages) | 766s/page (page 1 only) |
| Image processing (1700x2200 PNG) | 26.5s |
| VRAM peak | 6.76 GB |
| **Markdown output** | **VACÍO — 0 caracteres** |

**Diagnóstico:** PaddlePaddle cu129 carga el modelo en GPU y ejecuta forward pass,
pero la generación de texto (decode) no produce output. Probablemente incompatibilidad
sm_120 (Blackwell) con el kernel de generación de PaddlePaddle.

### Test 2: PyTorch/transformers directo

| Métrica | Resultado |
|---------|-----------|
| Model load (fp16, device_map=cuda) | OK |
| VRAM tras carga | ~1.8 GB |
| **Inferencia** | **CRASH** |

**Error:** `TypeError: create_causal_mask() got an unexpected keyword argument 'inputs_embeds'`

**Diagnóstico:** El modelo custom (`modeling_paddleocr_vl.py`) usa una API de
`create_causal_mask` incompatible con transformers 4.57.6. vLLM 0.18.0 actualizó
transformers de ~4.44 a 4.57, rompiendo la interfaz. El modelo fue probablemente
desarrollado con transformers 4.44-4.48.

### Test 3: vLLM como servidor

**No ejecutado.** PaddleOCR-VL-1.5 no está en la lista de modelos soportados de
vLLM 0.18.0 ([issue #33554](https://github.com/vllm-project/vllm/issues/33554)).

### Test 4: GGUF via llama.cpp

**No ejecutado.** llama.cpp no está instalado en WSL2. GGUF disponible en
[PaddlePaddle/PaddleOCR-VL-1.5-GGUF](https://huggingface.co/PaddlePaddle/PaddleOCR-VL-1.5-GGUF).
Esta es la opción más prometedora para la próxima sesión.

## Resumen de hallazgos

| Backend | Load | Inference | Output | Veredicto |
|---------|------|-----------|--------|-----------|
| PaddlePaddle nativo | OK | 26-766s | VACÍO | BROKEN |
| PyTorch/transformers | OK | CRASH | N/A | BROKEN (versión) |
| vLLM server | N/A | N/A | N/A | NO SOPORTADO |
| llama.cpp GGUF | N/A | N/A | N/A | PENDIENTE |

## Conclusión

**PaddleOCR-VL-1.5 NO está listo para producción en RTX 5090 (sm_120) con el
stack de software actual.** Tres backends probados, ninguno funciona:

1. PaddlePaddle nativo: output vacío (decode broken en sm_120)
2. PyTorch/transformers: API incompatible (transformers 4.57 vs ~4.44)
3. vLLM: modelo no soportado aún

## Opciones para próxima sesión (orden de prioridad)

1. **llama.cpp GGUF** — instalar llama.cpp con soporte sm_120, descargar
   PaddleOCR-VL-1.5-GGUF, ejecutar via llama-server. Mayor probabilidad de
   funcionar porque llama.cpp tiene soporte sm_120 confirmado.

2. **Pinear transformers** — `pip install transformers==4.44.0` (sin vLLM) y
   re-ejecutar test PyTorch directo. Riesgo: puede romper otros paquetes.

3. **MonkeyOCR-pro-1.2B** — basado en Qwen-VL, posiblemente más compatible
   con el stack actual. Probado en 4060 8GB.

4. **Mantener qwen2.5vl:7b** — el pipeline actual funciona. Priorizar Fase 5
   sobre la revolución OCR dado el deadline del Premio (4 mayo).

## Software instalado durante esta sesión (en WSL2)

```
pip install paddlex[ocr]==3.4.2    # VL extras
pip install vllm==0.18.0           # (conflictos nvidia-cublas con PaddlePaddle)
pip install accelerate==1.13.0     # Para device_map
pip install transformers==4.57.6   # (instalado por vLLM, puede causar problemas)
```

**ADVERTENCIA:** vLLM instaló nvidia-cublas-cu12==12.8.4.1 pero PaddlePaddle
necesita 12.9.0.13. Esto puede afectar PaddleOCR GPU regular (PP-OCRv5).
Verificar en próxima sesión: `python3 -c "from paddleocr import PaddleOCR; ocr = PaddleOCR(); print('OK')"`
