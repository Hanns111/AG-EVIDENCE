# ADR-012: Benchmark PaddleOCR-VL-1.5 como reemplazo del VLM

- **Estado:** EN PROGRESO — Benchmark parcial, pendiente backend vLLM
- **Fecha inicio:** 2026-03-24
- **Autor:** Claude Code (Auditor)
- **Aprobado por:** Pendiente revisión Hans

---

## Contexto

El pipeline actual (v4.1.0) usa **qwen2.5vl:7b** via Ollama como VLM para extraer
campos de comprobantes escaneados. Rendimiento actual:

| Métrica | Valor |
|---------|-------|
| Tiempo promedio/página | ~28s |
| Tiempo total DIRI2026 (21 páginas VLM) | ~10 min |
| VRAM | ~14.3 GB |
| Modelo | 7B parámetros |
| Precisión campos | Variable (RUC 0%, Fecha 31%) |

**PaddleOCR-VL-1.5** (0.9B, Baidu, enero 2026) reporta:
- 94.5% en OmniDocBench v1.5 (SOTA mundial)
- 8x más pequeño que qwen2.5vl:7b
- Mismo ecosistema PaddleOCR que ya usamos
- Apache 2.0, local-first compatible
- Soporte vLLM + llama.cpp (GGUF) desde marzo 2026

## Resultados del benchmark (2026-03-24)

### Test 1: PaddlePaddle nativo en RTX 5090

| Métrica | PaddleOCR-VL-1.5 (nativo) | qwen2.5vl:7b (Ollama) |
|---------|---------------------------|----------------------|
| Model size | 0.9B | 7B |
| Model load (cached) | **2.7s** | ~5s |
| Time/page (PDF 44p) | **766s** (INACEPTABLE) | ~28s |
| Time/page (imagen PNG) | **26.5s** | ~28s |
| VRAM peak | **6.76 GB** | ~14.3 GB |
| Markdown output | **VACÍO (0 chars)** | JSON con campos |

### HALLAZGO CRÍTICO: PaddlePaddle nativo + RTX 5090 sm_120 = BROKEN

1. **PDF processing absurdamente lento (766s/page)** — el pipeline layout+VL
   sobre PDF está corriendo en modo subóptimo, posiblemente CPU para layout detection
2. **Imagen individual: 26.5s** — comparable a qwen2.5vl pero con OUTPUT VACÍO
3. **El output markdown es consistentemente vacío (0 chars)** en ambos modos
4. **Diagnóstico probable:** PaddlePaddle cu129 + RTX 5090 sm_120 tiene incompatibilidad
   en la generación de texto (forward pass OK pero decode broken). Similar al problema
   documentado con MonkeyOCR en ADR-011.

### Test 2: vLLM backend (PENDIENTE — próxima sesión)

vLLM 0.18.0 ya está instalado en WSL2. Pendiente ejecutar:

```bash
# Opción A: PaddleOCR con backend vLLM
paddleocr genai_server --model_name PaddleOCR-VL-1.5-0.9B --backend vllm --port 8118

# Opción B: vLLM directo con modelo HuggingFace
python -m vllm.entrypoints.openai.api_server \
  --model PaddlePaddle/PaddleOCR-VL-1.5 \
  --port 8118

# Opción C: llama.cpp GGUF (descarga de PaddlePaddle/PaddleOCR-VL-1.5-GGUF)
# Requiere instalar llama.cpp primero
```

**NOTA:** vLLM 0.18.0 instaló conflictos con PaddlePaddle GPU (nvidia-cublas versiones
distintas). Puede requerir venv separado.

## Arquitectura comparada

```
Pipeline actual (v4.1.0):
  PDF → PyMuPDF (texto) → regex parser → si falla → render imagen → qwen2.5vl:7b → JSON

PaddleOCR-VL-1.5 (propuesto):
  PDF → PP-DocLayoutV3 (layout) → PaddleOCR-VL-1.5-0.9B (recognition) → Markdown

  Nota: El output es Markdown, NO JSON estructurado. Requiere post-processing
  con regex para extraer RUC, fecha, montos, IGV, etc.
```

## Software instalado durante esta sesión

- `paddleocr 3.4.0` — PaddleOCR con soporte VL (ya existía, upgrade a VL extras)
- `paddlex[ocr] 3.4.2` — Dependencias OCR extras (tokenizers, sentencepiece, etc.)
- `vllm 0.18.0` — Backend de inferencia recomendado para NVIDIA GPUs
- Modelo descargado: `~/.paddlex/official_models/PaddleOCR-VL-1.5/` (~1.8 GB)
- Layout model: `~/.paddlex/official_models/PP-DocLayoutV3/`
